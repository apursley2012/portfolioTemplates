<details open><summary>

## Purpose

</summary>

The **Portfolio Themes Templates** repository is the main directory for a collection of standalone static website themes. Each theme is intended to be published in its own GitHub repository, but this main repository works as the central index where users can browse the available designs, compare styles, view screenshots, and find the correct standalone repository for the theme they want to use.

This README is written for people who want to use one of the themes for their own website. It explains how the collection is organized, what each theme repository should contain, and how users should approach editing the themes. The focus is practical documentation, not a portfolio showcase.

</details>

---

<details open><summary>

## How This Collection Is Organized

</summary>

Each theme has its own folder in this documentation package and is also intended to have its own standalone GitHub repository. The standalone repository should contain the theme files, screenshots, assets, and a full README that explains exactly how to customize that specific theme.

This main collection repository should link to each standalone repository. Each standalone repository should also link back to the main collection so users can find related themes and future updates.

</details>

---

<details open><summary>

## Who Should Use These Themes

</summary>

These themes are intended for users who want a static website they can edit without setting up a database, backend, or complicated hosting stack. They can be used by developers, students, designers, artists, freelancers, writers, small business owners, and anyone who wants a personal or professional website that can be published through GitHub Pages.

</details>

<details open><summary>

## Repository Relationship

</summary>

Every theme should be treated as its own project repository. The main **Portfolio Themes Templates** repository is the directory and documentation hub, while each standalone repository is where a user downloads, edits, and deploys one specific theme.

Recommended relationship:

```text
Portfolio-Themes-Templates/
├── README.md
├── links to standalone theme repositories
└── collection-level documentation

individual-theme-repository/
├── index.html
├── about.html
├── projects.html
├── contact.html
├── components/
├── css/
├── js/
├── images/
└── README.md
```

</details>

# Using a Theme

<details open><summary>

## Choosing a Theme

</summary>

Users should choose a theme based on the type of website they want to create, the tone they want visitors to feel, and how much content they need to include. A developer may prefer a technical or terminal-inspired theme, while a designer may prefer a colorful or creative layout. Someone who wants a playful personal site may prefer one of the retro, fantasy, or operating-system-inspired themes.

</details>

---

<details open><summary>

## Editing a Theme

</summary>

After choosing a theme, users should open that theme’s standalone README and follow the customization guide. The most important edits usually include the site owner name, biography, project details, contact information, images, colors, and social links. Each individual README explains where those items are located and what to change.

</details>

---

<details open><summary>

## Publishing a Theme

</summary>

Each theme is intended to work with GitHub Pages. Users should place the theme files in a standalone repository, confirm that `index.html` is in the root, and enable GitHub Pages from the repository settings. After publishing, users should test every page, every navigation link, and every image path.

</details>

---

# Documentation Standards

<details open><summary>

## Individual Theme READMEs

</summary>

Each individual theme README should explain the purpose of the theme, show screenshots, describe the page and component structure, and tell users exactly what they need to change to make the theme work for their own life, career, projects, or business. The README should be detailed enough that a beginner can edit the theme without needing to reverse-engineer the files.

</details>

---

<details open><summary>

## Screenshots and Images

</summary>

Each theme should include at least one preview image. Screenshots should use readable file names that identify the theme and preview number. Users should replace the preview images after customizing the site so the README shows the current version of the theme.

</details>

---

<details open><summary>

## Customization Instructions

</summary>

The documentation should tell users where to change names, biographies, project information, contact links, social profiles, images, colors, fonts, and JavaScript-powered behavior. Instructions should point to specific files and explain what each change affects.

</details>
